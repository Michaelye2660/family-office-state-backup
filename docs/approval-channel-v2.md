# 批准通道v2·副账号方案（2026-07-12设计·CGM起草·**待委托人建号后启用**）

## 要解决的问题
approvals.md手动网页编辑安全但繁琐（EXT🔴6的解法v1）。需求:**既安全（GM/CGM均不可伪造）又自动（委托人只做最小动作）**。

## 核心事实（本方案的安全性来源）
GM与CGM在GitHub上都以**委托人主账号**行事——凡主账号API能做的（commit/评论/PR/审批），两个AI理论上都能做,故主账号的任何API动作都不构成委托人亲为的证明。**唯二不可伪造物**:①web-flow签名commit（须浏览器登录会话,API无法取得GitHub的GPG签名——注意核验时必须查"签名verified",不能只看committer名,后者API可任意填写）;②**一个AI双方均无凭据的第二身份**。方案v2用第②种。

## 方案设计（副账号批准通道）
1. **委托人建副账号**（一次性,手机2分钟）:注册第二个GitHub账号,如 `michaelye-approver`;主账号邀请其为本仓库collaborator（**Read权限即可**——私库Read可评论issue,不可push,权限面最小）。副账号凭据只存在于委托人手机,永不进任何AI对话或环境变量（N8同族纪律）。
2. **常设判断门issue**（CGM建,一次性）:标题「判断门·批准登记（常设）」,置顶说明格式。
3. **批准动作（每次约10秒,GitHub手机App即可）**:委托人以**副账号**在该issue下评论一行,如:`批准: L7生效+19c选项b暂缓` 或 `否: XXX`。评论≠编辑文件,手机App原生支持,零摩擦。
4. **自动转录（CGM侧,零人工）**:CGM在每次收件/班次运行时拉取该issue新评论,机械核验 **评论作者===副账号** 后,自动转录进 docs/approvals.md（附评论permalink+评论ID+时间戳）。**权威凭证=issue评论本身**（平台归属于副账号,AI无凭据仿冒）;approvals.md降为索引副本。
5. **审计规则同步**:L7与EXT核验"委托人亲为"时,判据=副账号评论（或web-flow签名commit,两者并行有效）;主账号API动作不再单独作为亲为证明。

## 伪造抵抗分析（诚实边界）
- GM/CGM以主账号评论→作者显示主账号,机械可辨,不构成批准 ✓
- GM/CGM无副账号凭据→无法以副账号发声 ✓
- 剩余风险:委托人手机失守=副账号失守（与一切个人凭据同级,超出本系统边界）;GitHub平台本身作恶（不设防,与git正本假设同级）。

## 启用条件
委托人回复副账号用户名→CGM建常设issue+邀请协作说明→委托人接受邀请→首条测试评论→CGM验通后〔M〕落档启用,approvals.md头部注记通道升级。**启用前,现行"定期输入"web-flow方式照旧有效。**
